# Specialized agents (brainstorm on top of my head) - some might be better as skills just throwing stuff

- Manager - For managing the state and documentation - controling the whole workflow)
- Brainstormer - takes a rough suggestion and aims to find good ways to implement. His job is to go wide rather than deep. Examines different possible ways to do something. Talks with the user gather data, finds best practices evaluates pros and cons. 
- Specifacator - good at finding the gray areas and good at asking the right questions to elaborate an idea. Questions should be configurable (no/few/regular/many). Default is regular. His job is not to suggest but rather to fill in to find missing pieces that are crucial and can lead to issue if left vague. Goes deep
- Architect - is able to pick out the best practices and design patterns to produce a suggestion on what the implementation shall be. Can summon the researcher. Suggest design patterns, librarires, abstractions. Doesn't go into detail but rather steers to ensure proper mechanisms are used. Goes through the code to see what is the current state and what is applicable but is not as deep as planner.
- Researcher - makes in-depth research online. Can be used by various other agents and skills.
- Planner - can use architects output to create a plan. The plan is actually rooted in the code and suggests specifics. The planer is good at two things. Being specific about implementation, splitting it deliverable parts
- Coder - Implements the plan. Good at coding
- Verifier - Receives the plan and checks the code. Good at finding discrepancies
- QA - Good at writting tests based on the plan
- Code Reviewer - reviews the code. Good at finding bad practices. (maybe it should be split into severl sub-agents for maing reviews focus on different aspects)
---- Bug hunter
---- Architect
---- TypeScript/JAva reviewer
---- ETC.
- Extender - used to extend the framework, knows how it works, how to write appropriate skills, etc.
- Debugger - adept at doing through debugging

For instance for my personal projects I have some, but for my work related we have a big list of java skils that are paramout for the projects we work on. - I imagine most agents to have --lite parameter. For instance imagine if the debug agent is very heavy maing hypothesis and verifying but I want to debug something quite straight forward but ** want to use the best practices without the ceremony ** I will use it in --lite mode. This is the keyword - keep guidelines/best-practices do not apply rigid ceremony
- I imagine the workflow to maintain a good technical documentation and a good product regarding the project. The product documentation is more valuable if we are doing a whole project as we define it. If we do just features in a big mature project then the product documentation shall be generated once and more rarely updated as usually features are not that big that they actually change the idea of the project (e.g. adding a new endpoint or a new field is something that is documented by the code itself - the documentaton shoul contain just the actuall use-case of the project to give a general idea). Rather if we build a new project this documentation is very important as we currently shape nad guide it (maybe that shall be a setting of the workflow something like vibe (vibe is vibe code - e.g. quickly spining big project) vs. maintain (this is usually for work project that I just contribute to)). The tech documentation is more or less the same but maybe less affected by that setting (vibe vs. maintain). The idea of the technical documentation is to document, how we work with the repo, how we structure APIs, how we structure converters/mappers, what we abstract, which design patterns we use, etc. The idea is to keep the project coherent and extend that documentation as the project grows. This should happen auto (hook or skill or each agent/skill referencing a common skill that describes it explicitly - dunno).
